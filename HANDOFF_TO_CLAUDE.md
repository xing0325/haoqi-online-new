# 好奇 Online：给 Claude 的完整交接书

> 这不是 Notion 复刻项目。请先完整读完本文件再改代码。
> 当前项目是前端原型，不是“已经接好后端的产品”。不要把示例数据、按钮或文案误判为真实业务状态。

## 0. 你在帮谁做什么

**产品名：** 好奇 Online
**一句话：** 把好奇学习社区的课程、正在发生的事、协作项目、阅读、活动和日常连接，变成一个轻巧、鲜活、可留痕的线上共同空间。

它有三种身份：

1. **成员的日常工作台**：今天上什么、课程有什么更新、谁在做什么、我该去哪里。
2. **社区的记忆库**：课程资料、项目过程、照片、作品、讨论不再散落于聊天记录、相册和临时 Notion 页面。
3. **对外可展示的窗口**：未来游客页面可让期末展观众、家长、潜在新成员理解“好奇平时到底怎么生活”。

核心原则：**网站应该像一个有人在生活的社区，而不是一套冷冰冰的学校管理系统。**

## 2. 已确定的产品信息架构

### 2.1 当前六个主空间

| 空间 | 给用户解决的问题 | 当前原型状态 | 后续真实能力 |
| --- | --- | --- | --- |
| 此刻（首页） | 今天我该去哪、有什么新事、社区有什么正在发生 | 已完成视觉和基本点击交互 | 聚合真实课程、动态、通知、状态与待办 |
| 课程 | 每门课有独立、可预览的动态空间 | 已有入口与卡片 | 课程资料流、内容页、评论、附件、课程成员 |
| 我的一周 | 官方课表与个人规划如何并存 | 已完成官方周表视觉 | 官方视图、个人视图、隐私、选修映射、调课通知 |
| 信用积分 | 出勤、请假、积分记录可解释可追溯 | 已完成积分/签到/请假草稿原型 | 真正签到、审批、规则、流水与管理员处理 |
| 阅读联赛 | 共同读书、留下摘抄和进度 | 已完成书架/进度/摘抄视觉 | EPUB 阅读器、书目、阅读时长、联赛与讨论 |
| 做事空间 | 项目把想法推进成成果 | 已完成项目总览与看板视觉 | 项目、任务、进度、文档、附件、表扬/协作 |
| 大家在干嘛 | 每日话题、投票、找搭子、活动 | 已完成社区大厅原型 | 状态流、讨论、投票、活动发起/报名、权限 |

### 2.2 首页是产品的心脏

首页不是“所有功能入口的九宫格”，而是一张当下的社区切片。它应始终优先呈现：

1. **今天**：课表、即时调课、即将开始的事。
2. **正在发生**：课程最新动态、同学正在做什么。
3. **我自己的节奏**：积分、阅读、个人计划。

新模块如要进入首页，必须回答：它是否帮助成员理解今天？如果只是一项后台指标或功能入口，不要占首页最醒目的位置。

### 2.3 课程空间的关键交互（非常重要）

课程不应是 Notion 那样一堆“只能看标题或第一行”的卡片。正确方向：

- 顶部为可横滑的课程头像栏；有更新的课程靠前，并有新动态标记。
- 下方是**按更新时间排序的课程综合动态流**，用户进课程前就能看见最近发生什么。
- 点课程头像进入该课程主页；课程主页有头像、名称、简介与内容瀑布流/标题流。
- 课程内容页应支持 Markdown、图片组、PDF、文档、表格和外链预览；不要强迫所有内容伪装成图文笔记。
- 课程卡片仍可从官方课表直达。调课应通过卡片标红/横幅提示，不要用“替换课程实体”的方式使用户失去课程入口。

### 2.4 官方课表与个人日程的边界

- 官方课表是公共事实：课程时段、地点、临时变化，成员不能改。
- 每个学生可创建自己的并行视图：删掉自己不去的课，或把时段留给别的事。
- 个人视图仍按好奇既有的课程时段切块；**第一版不要做任意时刻自由拖拽的日历**。
- 个人视图有公开/私密开关。
- 选修第一版可先显示“大选修/小选修”，之后再按选课结果映射到个人课程名。

### 2.5 内容与协作的权限模型

先把“谁能看到、谁能改”设计清楚，再碰实时协作：

- 所有内容属于一个明确的空间：课程、项目、活动或社区话题。
- 单人内容：作者编辑；其他人评论；草稿可公开预览与否由作者/空间规则决定。
- 协作内容：创建者可邀请编辑者；多人可以看到中间状态；评论与编辑权限分离。
- 图片多作者同时编辑很容易造成冲突：第一版采用“所有协作者可上传到素材池，指定内容负责人挑选和排序主图”的方式。
- 绝不实现“任何人都能随便改任何页面”的默认权限。

## 3. 视觉与体验：这是不可随意推翻的判断

### 3.1 当前的视觉方向

项目当前不是典型 SaaS 后台风格。整体应该是：

- **纸张感、轻松、有人味**：暖白背景、海军蓝文字、低饱和但有生命力的黄/珊瑚/浅蓝/薄荷绿。
- **带一点玩心，但不幼稚**：不规则圆形、轻微旋转、手帐式小标签、真实项目和人的状态。
- **信息很密，但呼吸充足**：小字号辅助信息、显著标题、明确网格、卡片不要过度阴影。
- **更像社区编辑页，不像企业 KPI 面板**。

已使用的语义色：

| 色彩 | CSS 变量 | 主要含义 |
| --- | --- | --- |
| 深蓝 | `--ink` / `--navy` | 主文本、导航、重点操作 |
| 黄色 | `--yellow` / `--lemon` | 共同好奇、积分、强调、阅读 |
| 珊瑚色 | `--coral` | 变更、行动、温度 |
| 浅蓝 | `--blue` | 阅读、课程、冷静的信息块 |
| 薄荷绿 | `--mint` | 完成、协作、松弛的状态 |

### 3.2 不要做的视觉改动

- 不要整体换成黑白/渐变玻璃/赛博朋克风。
- 不要引入一套臃肿组件库后把现有卡片全部抹平。
- 不要把中文标题改成巨大的无意义英语 hero 文案。
- 不要为了“统一”移除不规则形状、轻微旋转和局部手工感。
- 不要把每个页面都塞入同样的「数据卡 + 表格 + 右侧操作栏」模板。

### 3.3 质量底线

- 窄屏下导航可展开，内容不能横向溢出（看板/宽表可以在局部横向滚动）。
- 交互元素必须能聚焦、可点击；图标按钮应有 `aria-label`。
- 不能只做静态视觉：新增按钮若没有功能，至少应明确地打开草稿、显示状态或标注“尚未接后端”。
- 任何可写入真实数据的按钮要有成功/失败/加载状态；不要假装提交成功。

不要声称这些能力已经完成；不要把样例姓名、积分、日期和活动当作真实数据。

## 5. 推荐的下一阶段：先让它“可用”，再让它“无所不能”

### Phase A：工程基线（优先级 P0）

目标：不改变已确认的视觉与信息架构，把静态样机变成可维护的应用骨架。

推荐工作：

1. 先确认是否要迁移到 TypeScript + React/Vite 或保留原生 JS。**未经确认不要大爆炸式重写。**
2. 如果迁移，先做一个只迁移首页与导航的窄切片，确认视觉无回归后再迁其他页面。
3. 将当前硬编码示例数据抽到单独的 mock 数据层：课程、动态、成员状态、项目任务、积分记录。
4. 建立路由/页面边界，避免一个几千行 HTML 文件无限增长。
5. 为关键状态准备统一类型：加载中、空状态、权限不足、出错、草稿、已发布。
6. 加入最小质量门槛：格式化、静态检查、移动端截图或人工验收清单。

**验收：** 首页、课程、课表、积分、阅读、项目、社区在桌面和窄屏不发生明显视觉退化；浏览器前进/后退与地址可恢复到相应空间。

### Phase B：最小真实数据闭环（优先级 P0）

目标：至少让“课程动态”和“个人日程”从示例内容变成登录成员可使用的数据。

建议最小领域对象（名称可调整，关系不要丢）：

```text
User
  id, display_name, avatar_url, role(student|teacher|admin), profile_status, created_at

Term
  id, name, starts_on, ends_on, is_active

Course
  id, term_id, name, short_name, avatar_url, description, owner_id

CourseMembership
  course_id, user_id, role(member|teacher|assistant)

ScheduleSlot
  id, term_id, weekday, starts_at, ends_at, course_id, slot_kind(required|large_elective|small_elective|free)

ScheduleChange
  id, slot_id, occurs_on, change_type(location|time|cancelled|note), message, published_by, published_at

PersonalPlan
  id, user_id, schedule_slot_id, title, visibility(private|community), is_hidden

Post
  id, space_type(course|project|community), space_id, title, body_markdown, author_id, status(draft|published), published_at

PostAsset
  id, post_id, storage_key, asset_type(image|pdf|document|spreadsheet|link), sort_order

Comment
  id, post_id, author_id, body, created_at
```

关键产品规则：

- 课表更新是 `ScheduleChange`，不覆写 `Course`。
- 个人计划是覆盖/补充层，不篡改公共 `ScheduleSlot`。
- 所有可编辑内容记录 `author_id` 与时间，权限检查在服务端完成，不只靠前端隐藏按钮。
- 教师与管理员的“能编辑”不等于“能看到所有私密个人计划”；隐私规则要单独确认。

**验收：** 两个测试账号能看到不同的个人课表；教师发布课程动态后，成员首页能看到；调课能在首页提示和课表局部呈现。

### Phase C：信用积分与出勤（优先级 P1，但规则设计必须先行）

现有设想：每节课学生处于默认、签到、请假、缺席其中一种状态；请假在课程前 10 分钟关闭，老师审批；未签到且未请假会触发缺勤/扣分。

请在开始编码前确认下面这些政策问题，不要擅自填空：

1. 各类迟到、早退、病假、事假、老师手动豁免具体如何计分？
2. 扣分是否需要申诉？谁能撤销或修正？
3. 是否允许管理员手动新增奖励分？奖励规则是否公开？
4. 签到后是否允许撤销？课程结束多久后结算缺席？
5. 哪些人有权查看个人完整积分流水？排行榜是否真的需要？

建议的领域对象：

```text
AttendanceSession
  id, course_id, schedule_slot_id, occurs_at, opens_at, closes_at, status(scheduled|open|closed|settled)

AttendanceRecord
  id, session_id, user_id, status(pending|checked_in|leave_requested|leave_approved|leave_rejected|absent|excused), checked_in_at

LeaveRequest
  id, attendance_record_id, reason, submitted_at, reviewed_by, reviewed_at, decision

CreditLedger
  id, user_id, delta, reason_code, reason_text, related_type, related_id, created_by, created_at
```

签到反代签的现实结论：**四位数口令本身不可靠，学生互传就失效。**低复杂度方案可用短时效动态二维码/验证码，再配合校园 Wi‑Fi 或老师现场抽查；如要提高成本，需要近场蓝牙/NFC。技术不能完全消灭代签，制度和随机检查仍然必要。不要承诺“100% 防代签”。

**验收：** 任何一次签到、请假审批、自动缺席结算都会生成可追溯记录；同一事件重复执行不会重复扣分；学生只能读自己的记录，教师只处理自己课程。

### Phase D：课程内容与项目空间（优先级 P1）

先后顺序：

1. 课程动态发布/阅读/评论。
2. 内容附件与素材池。
3. 项目、成员、任务、看板状态。
4. 项目时间线/甘特图（不要在没有任务和日期模型前先画一张假的甘特图）。
5. 多人协作编辑与版本历史。

推荐的项目对象：

```text
Project
  id, title, summary, cover_asset_id, visibility, owner_id, status, starts_on, target_date

ProjectMember
  project_id, user_id, role(owner|editor|member|viewer)

ProjectTask
  id, project_id, title, description, column(idea|doing|done), assignee_id, due_at, sort_order

ProjectUpdate
  id, project_id, body, author_id, created_at
```

**验收：** 一个成员能发起项目、邀请成员、创建任务、把自己的任务从“正在做”移到“长出来了”；其他成员能看到进度但无权改不属于自己的任务。

### Phase E：阅读、活动、社区（优先级 P2）

这些很有魅力，但要建立在身份、内容、文件与通知基础上。

- 阅读：先实现书目、阅读进度手动记录和摘抄讨论；EPUB 内置阅读器放到后面。
- 社区状态：默认短状态、时效性强；应有过期、删除和举报策略。
- 投票：先做单选、截止时间、是否匿名、是否允许改票四件事。
- 活动：先做免费活动的发起、审核、报名、名额；收费、高门槛活动需要支付、退款、风控和责任边界，绝不能顺手做。

## 6. Claude 可以直接领取的任务单

每个任务单必须独立分支、独立提交、独立验收；不要让多个 Claude 同时改同一个大型文件。

### 任务 A：前端工程化评估（不直接重写）

**输入：** 当前仓库与本交接书。
**输出：** `docs/architecture-options.md`。

请比较“保持原生静态前端”和“迁移 TypeScript + React/Vite”的维护成本、迁移路径、风险与推荐结论。必须给出最小迁移切片。**不要改 UI 或代码。**

验收：一页说明能让负责人选择，且指出怎样保证当前视觉不回归。

### 任务 B：领域模型与权限矩阵

**输入：** 本文件第 5 节。
**输出：** `docs/domain-model.md` 与 `docs/permissions-matrix.md`。

写清成员、老师、管理员、游客对课程、课表、个人计划、动态、评论、项目、积分、请假的读取/创建/编辑/删除权限。标出所有尚需负责人决策的规则。**不要写数据库迁移或接入真实账号。**

验收：任何“谁能改什么”的问题能在文档中定位；不把前端隐藏当作权限。

### 任务 C：静态前端质量审计

**输入：** 当前 HTML/CSS/JS。
**输出：** 一个只包含可验证问题的审计清单与小修复 PR。

检查：窄屏溢出、键盘焦点、色彩对比、死按钮、URL hash、语义化结构、页面刷新状态。不得全量重写 CSS，不得更换视觉方向。

验收：修复范围小；有实际验证步骤；不因“优化”删掉现有设计特征。

### 任务 D：课程空间数据驱动原型

**前置：** 任务 A 的技术选择已被负责人确认。
**范围：** 仅课程总览、课程头像栏、综合动态流、课程详情、内容详情的假数据驱动版本。

验收：保留“课前可预览课程更新”的核心交互；至少有加载、空状态和无权限状态；不实现真实后端即可验证界面。

### 任务 E：出勤规则规格书

**输出：** `docs/attendance-spec.md`。

要求绘制状态机、列出时间边界、幂等与撤销规则、审批流程、人工干预入口、异常处理、审计需求。不要先写签到 UI，更不要实现所谓“防作弊黑科技”。

验收：开发者可按规格实现，老师可以读懂并指出哪条校规需要修改。

## 8. 协作与 Git 纪律

- 一个 Claude/一个人一次只负责一个模块；不要并行修改 `index.html` 和全局样式同一区块。
